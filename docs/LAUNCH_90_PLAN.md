# HantaWatch 90 分上线目标

## 产品目标

HantaWatch 的核心目标不是展示单个事件，而是形成可持续的数据闭环：

1. 面向中国国内受众，提供每日全球汉坦病毒公开信息监测、风险预警与解释。
2. 面向有出国需求的人，提供按国家查询的风险状态、最新事件、来源新鲜度和不确定性说明。

加拿大、瑞士、台湾省等都只是样例。系统必须能处理未来任何类似事件：官方通报、初筛阳性、待确认、输入病例、新闻线索、地方卫生部门公告、WHO/ECDC 延迟收录等。

## 90 分定义

达到 90 分不是指“没有遗漏”，而是指系统具备上线所需的核心可靠性：

| 能力 | 目标 |
|------|------|
| 每日更新 | collector 每天生成统一风险快照、国家风险快照和最新事件 |
| 国家查询 | 国家页不只显示静态基线，还显示近 30/90 天事件、信号热度和数据新鲜度 |
| 证据分层 | 官方通报、人工维护、新闻线索、AI 翻译内容分层展示 |
| 类似事件入链 | 任意国家出现类似新事件时，可通过自动 watcher 或手工事件进入首页、国家页和风险摘要 |
| 可解释 | 用户能看到来源、更新时间、确认状态和局限说明 |
| 不误导 | 页面避免把“未抓到”表达成“无风险”，并明确不替代官方旅行/医学建议 |

## 数据链目标形态

```text
WHO DON / ECDC / official watchers / Hantaflow / manual leads
  ↓
recent-cases-intl.json + mv-hondius-imports.json + country-signals.json
  ↓
country-risk-snapshot.json
  ↓
Web 国家页 / MiniApp 国家页 / 首页摘要 / API
```

## P0：上线前必须完成

### 1. 国家风险实时化

生成 `country-risk-snapshot.json`：

- 每个国家合成基线、输入事件、近 30 天/7 天信号、近 90 天事件。
- 输出国家级风险：`watch`、`elevated`、`active`、`baseline`。
- 输出 `latestEvent`、`evidenceLevel`、`freshness`、`statusZh`、`riskSummaryZh`。
- 国家页使用该快照，而不是只依赖手工基线。

### 2. 通用事件入链

类似加拿大的新事件出现时，不应需要改页面代码。

可接受入口：

- 官方 watcher 自动抓取。
- 手工 `news-leads-manual.json` 补录。
- Supabase 后台覆盖。

入链后应至少影响：

- 首页最新通报。
- 国家页对应国家卡片。
- 国家排序。
- 数据 freshness。

### 3. 来源 freshness

页面需要展示：

- 最近事件日期。
- 最近来源抓取时间。
- 信号最后出现时间。
- 是否官方 / 人工 / 新闻线索。
- 数据是否陈旧。

### 4. 保守上线话术

正式上线前使用：

- “全球汉坦病毒动态观察 Beta”
- “公开来源每日监测”
- “辅助了解，不替代官方医学/旅行建议”

避免直接宣称：

- “实时预警系统”
- “覆盖所有国家”
- “无事件即无风险”

## P1：一周内增强

- 增加官方 watcher 注册表，优先覆盖 WHO、ECDC、US CDC、PHAC/B.C. CDC、UKHSA、RKI、Santé publique France、KDCA、日本 NIID/厚劳省、台湾省 CDC、智利/阿根廷卫生部。
- 给每个 watcher 输出 `sourceLastCheckedAt`、`sourceLastUpdatedAt`、`ok/error/stale`。
- 增加后台审核队列：待确认、已收录、已隐藏、已归档。
- 增加国家详情页，展示最近 30/90 天事件时间线。

## P2：90 分以上

- HPI 与国家风险联动回测。
- 目的地订阅提醒。
- 重大事件推送。
- 规则配置化与审计日志。
- 多源交叉验证。

## 当前实施原则

1. 不为某个国家写死逻辑。
2. 所有新事件必须通过统一事件结构进入系统。
3. 国家页必须显示不确定性和来源新鲜度。
4. 新闻线索只能作为低证据等级信号，不能直接等同官方病例。
5. 自动化失败时保留旧数据，但必须在 meta/freshness 中可见。
