# 每日简报智能化修复日志

> 日期：2026-05-20 | 目标：让简报从"仪表盘文字版"升级为"流行病学编辑的每日摘要"

## 修复原则

1. 简报每天必须不同 — 优先展示变化，无变化时诚实说明
2. 简报 JSON 自描述 — 不依赖前端运行时计算
3. 简报与时间线差异化 — 简报回答"意味着什么"，时间线回答"发生了什么"
4. 轻量采集也能刷新简报 — 消除自相矛盾

---

## 修复记录

### Fix 1: cluesLast24h 写入真实值

**文件：** `services/collector/hantawatch_collector/builder.py`  
**改动：** `build_daily_brief` 新增 `clues_last_24h` 参数（默认 0），替代硬编码 `"cluesLast24h": 0`。  
**效果：** JSON 中的 `cluesLast24h` 现在是真实线索计数，不再永远是 0，API 消费者可直接读取。

### Fix 2: feeds-only 刷新 daily-brief.json

**文件：** `services/collector/main.py` + `.github/workflows/collect-data-light.yml`  
**改动：** `_run_feeds_only` 结束后读取现有 `daily-brief.json`，更新 `cluesLast24h` 和 `headline24h`；CI workflow 增加 `daily-brief.json` 到 git add 列表。  
**效果：** 每小时轻量采集后，简报的线索计数和 24h 要点保持同步，不再自相矛盾。

### Fix 3: MiniApp hondiusImportSummaries

**验证：** `apps/miniapp/src/lib/data.ts` L230 已存在该导出，无需修改。

### Fix 4: AI 增强层接收昨日简报

**文件：** `services/collector/hantawatch_collector/ai_brief.py` + `services/collector/main.py`  
**改动：** `enhance_daily_brief` 新增 `previous_brief` 参数；payload 中增加 `yesterdayBrief` 上下文；main.py 调用时读取现有 `daily-brief.json` 作为昨日简报传入。  
**效果：** LLM 现在知道昨天说了什么，可以判断信息是否真正新增，避免复读。

### Fix 5: 变化检测 — 动态 oneLine

**文件：** `services/collector/hantawatch_collector/builder.py`  
**改动：** `build_daily_brief` 中新增变化检测逻辑：`global_cases_delta`、`hpi_delta`、`days_since`、`distance_delta_km`、`clues_last_24h`。有变化时生成"今日变化：..."句式，无变化时诚实说明"今日无新增官方通报或监测信号"。同时保留 `structuralLine` 供 UI 使用。  
**效果：** `oneLine` 每天内容不同，用户不会连续看到同一句话。

### Fix 6: 行动建议 userActionHint

**文件：** `packages/shared/src/daily-brief-display.ts`  
**改动：** `BriefSectionContent` 新增 `userActionHint` 字段；`buildBriefSectionContent` 中基于 HPI + 输入病例状态生成行动建议。  
**效果：** 简报从现在起包含用户可行动的信息，而非纯数据展示。

### Fix 7: DailyBriefSection 与时间线差异化

**文件：** `apps/web/src/components/daily-brief-section.tsx` + `apps/miniapp/src/components/daily-brief-section.tsx`  
**改动：** 两端 DailyBriefSection 增加"行动建议"区块（绿色卡片），展示 `userActionHint`；与下方"最新通报"时间线形成差异化。  
**效果：** 简报回答"这意味着什么 + 我该做什么"，时间线回答"发生了什么"。
