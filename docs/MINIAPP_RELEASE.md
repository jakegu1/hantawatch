## 0 · 角色与前置条件

- **微信开发者工具**（最新稳定版）已安装并用**管理员/开发者微信**登录。
- 你的微信号在该小程序的 **mp.weixin.qq.com → 成员管理** 里有：
  - **开发者**权限 → 才能在工具里**上传**代码；
  - **管理员**（或运营者+提审权限）→ 才能在后台**提交审核 / 发布**。
- 一次完整链路：**本地构建 → 工具上传（开发版本）→ 后台提交审核（审核版本）→ 审核通过后发布（线上版本）**。

## 1 · 提交前必查（合规 + 配置）

> 这几项不对，要么真机报错，要么审核被拒。**第一次发布尤其要逐条确认。**

1. **request 合法域名**（最容易踩）
   - 小程序所有网络请求走 `https://www.bingduguancha.com`（见 `apps/miniapp/src/utils/api.ts` 的 `API_BASE`）。
   - 后台路径：**开发 → 开发管理 → 开发设置 → 服务器域名 → request 合法域名**，必须包含 `https://www.bingduguancha.com`。
   - ⚠️ 之前真机"离线 · 显示缓存"那类问题，根因常是这个域名没进白名单（现在该误导性提示已移除，但要让"实时叠加拉取"在真机生效，仍需白名单 + HTTPS + 有效证书）。
2. **用户隐私保护指引**
   - 后台：**设置 → 用户隐私保护 → 用户隐私保护指引**，必须填写并审核通过。
   - 我们基本不采集个人敏感信息（无登录、无健康问卷上报）；按实际填"不收集/仅匿名统计"即可。调用任何 `wx.getUserProfile` 等接口前必须先声明。
3. **服务类目**
   - 选**工具 / 资讯 / 科普**类。**不要选"医疗 / 健康 / 在线问诊"**等需资质（《医疗机构执业许可证》等）的类目。
   - 文案合规：不得自称"官方 / 疾控 / 权威"；数据写"来源：WHO / ECDC / 中国疾控"是**引用**，可以。
4. **基础信息**：小程序名称、头像、简介、客服/反馈入口齐全。
5. **本地校验全绿**：
   ```powershell
   pnpm --filter @hantawatch/miniapp exec tsc --noEmit -p tsconfig.json
   pnpm --filter @hantawatch/miniapp build:weapp
   ```

## 2 · 一步步发布

### 步骤 1 — 构建最新产物
```powershell
# 在仓库根目录 d:\Work\Hanta
pnpm --filter @hantawatch/miniapp build:weapp
```
- 成功输出 `Compiled successfully`，产物写入 `apps/miniapp/dist/`。
- ⚠️ 本机 Node 偶发 `VirtualAlloc failed`：**串行构建**，别同时跑多个 Taro/Next build。

### 步骤 2 — 在开发者工具打开并自检
- 开发者工具 → **导入项目** → 目录选 `d:\Work\Hanta\apps\miniapp` → 确认 AppID = `wxc26882b5496a039e`。
- 等待编译，在**模拟器**用 **375×667（iPhone SE）** 目测：
  - 首页顶部蓝色板块（头条 / 距离·HPI / 全球分布 / 官方风险 / 7 天趋势）；
  - 实时态势"全球分布"各国合计是否 = 表头确诊数；
  - "最近事件"时间线 + "监测中 / 隔离监测中"文案；
  - 来源行是纯文字（无"点击复制链接"）。

### 步骤 3 — 真机预览（强烈建议）
- 工具点**预览** → 管理员微信扫码 → 真机看一遍。
- 重点验证：`www.bingduguancha.com` 网络请求是否成功、字号/对比度、分享卡片标题。

### 步骤 4 — 上传（生成"开发版本"）
- 工具右上角点**上传**。
- 填：
  - **版本号**：语义化，如 `1.0.1`（见每次发布的"本次发布"小节）。
  - **项目备注**：本次改动摘要（见下方模板）。
- 上传成功 → 代码进入后台 **版本管理 → 开发版本**。

### 步骤 5 — 提交审核（开发版本 → 审核版本）
- 登录 **mp.weixin.qq.com** → 进入小程序 → **管理 → 版本管理**。
- 在"开发版本"找到刚上传的版本 → **提交审核**。
- 按表单填：功能页面、（如需）测试账号、类目、隐私指引、审核备注。
- 提交后进入**审核中**，通常几小时 ~ 1 个工作日。

### 步骤 6 — 发布（审核版本 → 线上版本）
- 审核**通过**后，在"版本管理 → 审核版本"点**发布**。
- 可选**灰度发布**（按比例分阶段放量），稳妥起见首次可直接全量。
- 发布后旧版本用户会在下次冷启动时更新到新版。

## 4 · 可选：CI 自动上传（miniprogram-ci）

适合以后自动化（首次手动更稳）：
1. 后台 **开发设置 → 小程序代码上传** 生成**上传密钥**，并把运行机 IP 加入白名单。
2. `pnpm add -D miniprogram-ci`（放在 miniapp 或专门的发布脚本里）。
3. 脚本示例（伪代码）：
   ```js
   const ci = require('miniprogram-ci');
   const project = new ci.Project({
     appid: 'wxc26882b5496a039e',
     type: 'miniProgram',
     projectPath: require('path').resolve(__dirname, '../apps/miniapp/dist'),
     privateKeyPath: process.env.MINIAPP_PRIVATE_KEY_PATH,
     ignores: ['node_modules/**/*'],
   });
   await ci.upload({ project, version: '1.0.1', desc: '…', setting: { es6: true } });
   ```
- `upload` 仍只生成"开发版本"，**提交审核/发布仍需在后台手动操作**（微信未开放提审 API）。

### v1.0.1 — 2026-06-03（待上传）

**最重要：数据改为运行时自动更新（修复"发布后数据冻结"）。**
此前小程序把采集器 JSON 在**构建时**打包进小程序包，发布后数据就固定不动——线上停留在 5-29 正是此因。本版改为运行时从
`https://www.bingduguancha.com/api/miniapp-snapshot` 拉取最新数据，并用同一套逻辑整体重算视图模型
（`DataProvider` + `useAppData`，见 `apps/miniapp/src/lib/data-provider.tsx` / `lib/app-data.ts`）。
发布本版后，数据在**每次冷启动 / 下拉刷新**时更新，**日常数据更新不再需要反复发版**（只有改代码才需再发版）。

- **前置（重要）**：本版依赖新接口 `GET /api/miniapp-snapshot`。请**先确认 web 端（Vercel）已部署该接口**
  （已在 `main`，推送后自动部署），再上传小程序。即使顺序反了也不会白屏——拉取失败会安全回退到打包进去的基线数据，
  下次冷启动接口可用时自动恢复为最新。
- **同时修复**：首页「HPI 近 7 天趋势」之前只有当天 31、前 6 天 24（输入加成只作用于最后一个点）；现整条曲线都应用实时输入加成。
- **口径**：实时态势新增「口径说明」行；时间线中跨 UTC 零点的同国初筛阳性信号合并为一条（不再叠加成两条 +N）。
- **上传备注可粘**：`数据改为运行时拉取（修复发布后数据冻结）；修复 HPI 7 天趋势；实时态势口径说明`。
